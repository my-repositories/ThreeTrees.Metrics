Project Structure

Typical Project Structure

The structure for typical project is based on popular github repositories: ASP.NET MVC, NLog, EntityFramework.

.
|-- artifacts/
|-- libs/
|-- docs/
|-- scripts/
|-- src/
|-- tests/
|-- tools/
|-- .gitignore
|-- .gitattributes
|-- CHANGELOG.txt
|-- default.ps1
|-- INSTALL.md
|-- LICENSE.txt
|-- README.md
L-- TODO.txt
  • artifacts - HTML pages, SQL scripts, other stuff.
  • docs - Project documentation in Markdown or reStructuredText format.
  • scripts - PSGallery scripts and any other additional scripts.
  • src - Source code.
  • tests - Project tests. This can be separate assembly within src directory or separate project in tests.
  • tools - Any additional tools we need for project build like ILMerge.
  • CHANGELOG.txt - Changelog that is supported between versions. Can be sent to whole team and customer.
  • README.md - Info about the project, links to artifacts, google drive, postman, swagger, UI, specs, design files, etc.
  • default.ps1 - PSake tasks.
  • INSTALL.md - How to build, install, compile, publish (CI) project, how to do database migrations. This document should be enought for new developer to start working on project.
  • LICENSE.txt - TBD
  • TODO.txt - Any general to do items.

The structure of src folder:

.
|-- ...
|-- src/
|   |-- Company.TestProject.Abstract/
|   |-- Company.TestProject.Cli/
|   |-- Company.TestProject.DataAccess/
|   |-- Company.TestProject.Domain/
|   |-- Company.TestProject.Infrastructure/
|   |-- Company.TestProject.IntegrationTests/
|   |-- Company.TestProject.Web/
|   |-- Company.TestProject.Shared/
|   |-- Company.TestProject.Services/
|   |-- Company.TestProject.Tests/
|   L-- Company.TestProject.sln
L-- ...
  • Abstract - Optional assembly that should not reference any libraries or projects. Contains basic POCO classes used by application (commands, exceptions, DTOs, API requests and responses). This assembly can be shared to other applications.
  • Cli - Command line interface. Various handlers can be here.
  • DataAccess - Assembly that contains database access logic. There should be implementation for Unit of Work, Unit of Work Factory, Repositories related to EntityFramework, NHibernate, raw SQL. The assembly references ORM directly.
  • Domain - Domain of application. Business logic goes here. See below.
  • Infrastructure - Optional component. Shared infrastructure details: dependency injection container setup, web components, WPF components and other application level stuff related to backend. You can have more than one infrastructure assembly. Think about it like the place where you can keep shared components across web or WPF projects. For example in Infrastructure.UI assembly you can keep common used UI controls that are used in Manager and Client WPF applications. Another example is Infrastructure.Mvc contains route extensions, API response class, filters related to ASP.NET MVC used across all web projects.
  • IntegrationTests - Integration tests.
  • Web - ASP.NET MVC application. Controllers, JavaScript, Frontend templates.
  • Shared - Helpers, extensions related to .NET go here. For example IEnumerable extensions or validation helpers. The project should not refer other projects.
  • Services - Services that relates to 3th party dependencies. If you do not want to "pollute" domain with dependencies to other libraries move them to separate assembly. For example that can be IEmailSender in domain and MailKitEmailSender implementation in services.
  • Tests - Unit tests.

Structure of Company.TestProject.Domain:

.
|-- ...
|-- src/
|   L-- Company.TestProject.Domain/
|       L-- Users
|           |-- Abstract
|           |-- Commands
|           |-- Dtos
|           |-- Entities
|           |-- Events
|           |-- Exceptions
|           |-- Handlers
|           |-- Queries
|           |-- Repositories
|           L-- Services
L- ...

In the root of domain there are bounded contexts. Bounded context is explicit boundaries in terms of application organization, logical data and processes separation. For example typical contexts may be Users, Products, Sales, Support, System. These contexts may refer to each other. Also the project contains interfaces for unit of work and unit of work factory.

  • Abstract - Interfaces and abstract classes go here.
  • Commands - POCO objects that describes processes within context. Commands should change application state (database CRUD). The commands just describe data that needs to implement action. For example CreateUserCommand, UpdateProductCommand, ChangeUserPasswordCommand, SetLockoutPeriodCommand. It is recommended to have Command suffix for every class. Actual operation is in Handlers namespace. Command should have one handler.
  • Dtos - Data transfer objects. For example UserBranchDto, InvoiceSummaryDto. It is recommended to have Dto suffix for every class.
  • Entities - Entites refer to the context. Usually they map to database tables. It is recommended to have data annotation attributes for fields. Every entity may have basic business logic refers to it.
  • Events - POCO classes that contain data needs for firing events. Event describes that something has happened. It is recommened to have Event suffix for every class. For example when user has been created (CreateUserCommand) the UserCreatedEvent is generated with new user id. Event may have several handlers.
  • Exceptions - Custom exceptions used within context. It should be domain exceptions (refer to DomainException in Saritasa.Tools.Domain).
  • Handlers - Commands and events handlers. Actual code and data manipulation. Within each handler there should be separate unit of work instance.
  • Queries - Common used queries. They use unit of work's database context to retrieve data from database. May be used to group LINQ queries that are used in separate places. May also contain some logic, may generate exceptions. For example UsersQueries, UsersSummaryQueries. It is recommended to have Query or Queries suffix for every class.
  • Repositories - Contains interfaces for repositories.
  • Services - Contains operations that are used by different commands, handlers. May be used to share business logic code.

You may move Exceptions, Commands, Dtos, Events may be separated to another assembly (Abstract). Later the assembly can be used for better integration with another applications.

  • Frontend - JavaScript, css, sass and templates. Frontend part of our web application.
  • Static/bundles - Bundles part of frontend application. Assemblies from bower and npm, generated by Gulp, WebPack. Should not be included to git.
  • Static - Assets, resources.

FAQ

Q: May domain project reference ORM library? A: The basic idea behind IUnitOfWork, IUnitOfWorkFactory is to cleanly separate business logic from ORM and any data storage. But in that way we are limited by only set of abstraction that are provided by repository and unit of work. For example we cannot use specific ORM features like async queries. So that on most cases it is possible to reference ORM to use these features. All raw SQL queries should be in repositories.

Q: Why do we need separate commands and handlers? A: First of all the abstraction allows to replace handler and provides much clean infrastructure. Also if Saritasa.Tools.Messages is used you can use these commands as JSON later for integration tests, enchanced logging, better debug.

Q: This is very similar to CQRS approach. A: Yes, but we do not strogly follow it. For example command can do read queries and return data within it, event sourcing is not used by default.

  • Understanding Onion Architecture

    Only the infrastructure layer knows about the database and data access technology (Entity framework or ADO.NET) and other layers don't know anything about from where the data comes and how it is being stored.

    The basic principle of Onion Architecture is to follow the boundaries of these layers – the inner layer cannot depend on its outer layer but can depend on layers beneath.
